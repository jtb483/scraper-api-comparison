## Google Search API 实际能做什么

ScraperAPI 的 Google Search 端点本质上是帮你绕过 Google 的反爬机制，把搜索结果页的数据以结构化 JSON 返回。我主要用它做三件事：

1. 批量查关键词排名，每天跑一次，监控自己站和竞品站在目标词上的位置变化
2. 抓取 SERP 特征元素，比如 People Also Ask、Featured Snippet、Local Pack 这些模块的内容
3. 做地域化搜索结果对比，同一个词分别拉美国、英国、日本的结果看差异

请求方式很简单，就是往 API 端点发一个 GET 请求，带上你的 API key、搜索词、国家代码这些参数。返回的 JSON 里会把自然结果、广告、相关搜索这些拆开给你。

我试过一个晚上跑 8000 个关键词的排名检查，用 Startup 套餐的 50 并发，大概 40 分钟跑完。没遇到封号或者大面积失败的情况。偶尔有几个请求超时重试就好了。

## 信用消耗机制：这个地方容易踩坑

ScraperAPI 的计费不是简单的"一个请求扣一次"。它有一个信用倍率系统，不同类型的请求消耗的信用不一样。

普通网页抓取算 1 个信用。但如果你开了 JavaScript 渲染（render=true），一个请求消耗 10 个信用。用 Google Search 端点的结构化数据抓取，每次消耗大概 10 到 25 个信用不等，取决于你请求的参数复杂度。

这意味着什么？你的 Hobby 套餐标称 100,000 次请求，但如果你全部用来跑 Google SERP 结构化抓取，实际能跑的次数可能只有 4000 到 1000 次。

我第一个月就是没注意这个，以为 100k 额度够用一个月，结果第 12 天就用完了。后来我调整了策略：能用普通模式抓的页面就不开 JS 渲染，SERP 抓取只在必要时用结构化端点，其他时候用原始 HTML自己解析。这样同样的额度能多撑两倍时间。

## 常见问题

**ScraperAPI 的 Google Search API 返回的数据是结构化的还是原始 HTML？**

两种都支持。你可以用它的 Structured Data 端点直接拿 JSON 格式的搜索结果，也可以用普通端点抓原始 HTML 自己解析。结构化端点省事但消耗信用更多，原始 HTML 省信用但你得自己写解析逻辑。

**免费套餐的 5000 次请求够做什么？**

如果只是测试 API 能不能跑通、返回格式对不对，5000 次绰绰有余。但如果你想用免费额度做正式的数据采集项目，基本不够。考虑到 SERP 请求的信用倍率，实际能跑的 Google 搜索可能只有几百次。

**请求失败会扣信用吗？**

不会。ScraperAPI 只对成功返回 200 状态码的请求计费。超时、被拦截、服务端错误这些情况不扣信用。这点我验证过，确实如此。

**支持抓取 Google 以外的搜索引擎吗？**

支持。除了 Google，它也能抓 Bing、Yandex 等搜索引擎的结果页。不过我主要用 Google 端点，其他的没深度测试过。

**可以指定搜索结果的国家和语言吗？**

可以。请求参数里有 country_code 和 language 字段，你可以精确控制从哪个地区的 Google 抓结果。我经常用这个功能做多地域排名对比。
